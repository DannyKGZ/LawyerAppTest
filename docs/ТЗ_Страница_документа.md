## 1. Что за приложение

**LawyerApp** — учёт истории правок юридических документов **с ответственностью**.

Юрист работает на аутсорсе для компании-клиента. Над одним договором работает много
людей со стороны клиента:

- **менеджеры** — договариваются с заказчиком об условиях;
- **бухгалтер** — правит цифры, реквизиты, условия оплаты;
- **старший юрист** — выверяет формулировки;
- **гендиректор** — финальные решения.

Каждый вносит правки в договор, и возникает набор проблем (это и есть **боль**, ради
которой делается продукт):

1. **Нет единой истории документа.** Непонятно, что было, что стало и почему.
2. **Аутсорс-юрист — «чёрный ящик».** Правки сваливаются на него без контекста —
   «пусть сам разбирается». Он не знает, о чём менеджеры договорились с заказчиком.
3. **Нет ответственного за правку.** Все скидывают юристу. При споре/суде некого
   спросить, почему в договоре появилась та или иная формулировка.
4. **Юриста делают крайним.** На правке настоял менеджер, но если потом будет суд —
   скажут «ты юрист, ты должен был думать». Сейчас это **выжигает юриста** (вплоть до
   желания уйти).

**Цель продукта:** простой источник правды по истории каждого договора, где у каждой
правки видно — **кто настоял, кто отвечает, почему внесли, чем подтверждено и какое
решение принято.** Это доказательная база на случай спора и снятие с юриста роли
«выгребной ямы».

## 2. Ключевые понятия (нужны, чтобы понять экран)

- **Проект** = пакет документов одного договора (основной договор + приложения +
  допсоглашения).
- **Документ** — один из этих файлов. У него есть **версии** (`v1 → v2 → v3`): загруженные
  снапшоты. Версия — это веха.
- **Правка (ЗРС)** — одно изменение пункта документа. У правки есть:
  - **пункт** (`clause`) — «п. 5.2 Срок оплаты»;
  - **было → стало** (`before` / `after`);
  - **ответственный/инициатор** (`responsibleId`) — **кто настоял** на правке (менеджер,
    бухгалтер, гендир). Это юридически важный человек;
  - **автор** (`authorId`) — кто технически оформил запись (обычно помощник юриста).
    `автор ≠ ответственный` — это принципиально;
  - **аргументация**, **дедлайн**, ссылка на **Битрикс**, **доказательство** (скрин),
    **приватная заметка юриста**;
  - **статус**: черновик · на рассмотрении · принято · на доработку · отклонено · внесено;
  - **уровень решения** (`tier`) — кто вправе принять (см. ниже).

### Уровень решения (`tier`) — второй смысловой стержень экрана

Помимо «кто ответственный», у каждой правки есть **уровень — кто вправе её принять**:

| Уровень | Значение |
|---|---|
| **Можно самим** (`self`) | низкий риск, юрист/помощник закрывает сам |
| **Решает ответственный** (`responsible`) | обычная правка, решение за инициатором |
| **Блокер · командой** (`blocker`) | **вне компетенции юриста.** Юрист **не решает в одиночку** — эскалация и защита командой |

Блокеры — это и есть «защита юриста»: граница, где нельзя вешать решение на одного
человека. На экране они должны быть **различимы сразу**.

### Жизненный цикл правки

```
Черновик → На рассмотрении →─┬─ Принято → Внесено в версию (vN+1)
                             ├─ На доработку → назад в черновик
                             └─ Отклонено → зафиксировано
```

История **append-only**: ничего не удаляется и не вычёркивается, даже отклонённое —
это часть доказательной базы.

## 3. Что должно быть на экране (данные)

Страница = **история одного документа**. На ней есть:

- **шапка**: название документа, проект/контрагент;
- **версии** (`v1…v3`) с возможностью скачать снапшот;
- **правки** — главный контент. Их может быть от нескольких до десятков.

Для каждой правки доступны все поля из §2. **Что из этого показывать в общем списке, а
что прятать в детали — решаете вы** (это и есть суть задачи про лаконичность).

### Особые ситуации, которые экран должен честно показывать

- **Дедлайн в прошлом → просрочка** (правка ещё открыта). Должно быть заметно.
- **Конфликт по одному пункту.** Двое настаивают на разном для одного и того же пункта.
  Реальный пример из данных (документ `d1`, пункт 7.1 «Неустойка»):
  - Петров (менеджер) настаивает снизить неустойку `0,1% → 0,05%` — **блокер**;
  - Орлов (ст. юрист) настаивает повысить `0,1% → 0,2%` — **блокер**.
  Это конфликт: на один пункт две открытые правки. Юрист не выбирает сам → команда.
- **Старая правка, которая «висит».** Заведена давно, ждёт решения ответственного
  (например, менеджер в отпуске) — она должна оставаться видимой как открытый риск, а
  не теряться.

## 5. Ограничения и принципы (важно соблюсти)

- **Лаконичность — главное требование.** Меньше текста и шума на первом экране. Деталь
  по правке — по клику (карточка/панель), а не вся сразу в списке.
- **Нейтральная палитра.** База — белый фон, серый текст (zinc), серые границы. Цвет —
  **дозированно и по смыслу**: emerald — решено/действует, amber — на рассмотрении/открыто,
  red — отклонено/просрочка/**блокер**, zinc — нейтрально/черновик.
- **Статус и уровень — маленький цветной кружок + нейтральный текст**, а не пёстрые
  заливки-плашки. Один набор иконок (lucide), без эмодзи-зоопарка.
- **Выровненные колонки / предсказуемая сетка** — ничего не должно «прыгать» и
  расползаться «лесенкой» при разном объёме данных.
- **Append-only:** отклонённое/на доработке не вычёркивать и не прятать — приглушить, но
  оставить видимым. История ценна.
- Используйте готовые примитивы из `src/components/ui/`, не изобретайте с нуля.

## 7. Как мы будем оценивать результат

Главный критерий — **за 5 секунд на странице человек понимает, что происходит с
документом.** Конкретно смотрим:

1. **Считываемость с одного взгляда.** Видно: на какой стадии документ, что открыто/в
   работе, что закрыто. Нет ощущения «стены текста».
2. **Ответственность видна.** По каждой правке понятно, **кто настоял** (а не «юрист сам
   придумал»). Это центральная ценность продукта.
3. **Блокеры и просрочка не теряются.** Сразу заметно, где правки вне компетенции юриста
   и где сорван срок.
4. **Конфликт по пункту читается как конфликт**, а не как две похожие строки.
5. **Лаконичность.** Детали спрятаны в один клик, первый экран чистый.
6. **Дисциплина дизайна.** Соблюдены принципы §5 (нейтральная палитра, кружки-статусы,
   ровная сетка, один набор иконок).
7. **Качество кода.** TypeScript без ошибок (`npx tsc --noEmit`), линт чистый
   (`npm run lint`), переиспользование примитивов, читаемые компоненты.
8. **Обоснование решения.** Короткая записка: какой главный вопрос экрана вы выбрали и
   почему раскладка устроена именно так.
